Overall Assessment

The Banarasi Kala backend is a well-structured, feature-rich Node.js/Express/PostgreSQL e-commerce API covering products, orders, payments (Razorpay), shipping (ShipRocket), wallet/referrals, and admin flows. The architecture is thoughtful — service/controller separation, Sequelize ORM, Cloudinary for images, Nodemailer for emails — and reflects solid engineering effort.

However, several critical production blockers must be resolved before launch: unprotected ShipRocket admin routes, absence of rate limiting, in-memory OTP storage, no pagination on list endpoints, and multiple business logic edge cases in the order/payment flow.

🔴 Launch Blocker: ShipRocket order management routes are completely unprotected — any person on the internet can cancel shipments, push orders to courier, or generate labels without authentication. This is the most critical issue in the entire codebase.
💡 Positive Highlights: The business logic is sophisticated — weighted shipping allocation per item, deduplication-keyed wallet transactions, COD-blocking fraud prevention, and referral reward scheduling with 7-day delay are production-grade patterns. The core foundation is solid; the gaps are fixable.

Section 4
Critical Security Vulnerabilities
These MUST be fixed before any production deployment
CRITICAL

BUG-001 — ShipRocket Routes Have Zero Authentication

✓ FIXED
File: src/routes/ShipRocketRoutes.js  |  Lines: 1–60  |  Impact: Full Shipment Sabotage

Every ShipRocket route — including push-to-courier, AWB assignment, label generation, manifest creation, cancellation, and return initiation — is publicly accessible with no authentication whatsoever. Any anonymous internet user can cancel live shipments, push random orders to couriers, or generate fraudulent return labels.

Fix: Add authMiddleware, adminMiddleware to all ShipRocket write routes.
// ShipRocketRoutes.js — CURRENT (broken) router.post('/push-order/:orderId', ShipRocketController.pushOrder); // FIXED router.post('/push-order/:orderId', authMiddleware, adminMiddleware, ShipRocketController.pushOrder );
CRITICAL

BUG-002 — No Rate Limiting on Authentication Endpoints

✓ FIXED
File: src/index.js, AuthRoutes.js  |  Impact: Brute-Force Attacks, OTP Guessing

Login, forgot-password, and OTP verification endpoints have no rate limiting. An attacker can attempt unlimited password guesses or brute-force 6-digit OTPs (only 1,000,000 combinations) without any throttling. At 100 requests/second, all OTPs are crackable in <3 hours.

Fix: Install express-rate-limit and apply per-route limits.
// npm install express-rate-limit const rateLimit = require('express-rate-limit'); const authLimiter = rateLimit({ windowMs: 15 * 60 * 1000, // 15 minutes max: 10, // 10 attempts per window message: { error: 'Too many attempts. Try again after 15 minutes.' }, standardHeaders: true, legacyHeaders: false, }); // Apply to auth routes in AuthRoutes.js router.post('/login', authLimiter, AuthController.login); router.post('/forgot-password', authLimiter, AuthController.forgotPassword); router.post('/verify-email-otp', authLimiter, AuthController.verifyEmailOtp);
CRITICAL

BUG-003 — OTP Stored In-Memory (Lost on Server Restart)

File: src/services/AuthService.js  |  Impact: Auth Bypass, Data Loss

Email OTPs are stored in a JavaScript Map in server memory. Every server restart, Render.com cold start, or deployment clears all active OTPs — users in the middle of registration get locked out. Additionally, this won't scale to multiple server instances. There is also no cleanup of expired OTPs, causing slow memory leak over time.

Fix: Persist OTPs to the database with an expires_at column, or use Redis with TTL. Run a cleanup job to remove expired rows.
// Option A: Database OTP table (simpler, no Redis needed) // Model: EmailOtp.js { email: { type: DataTypes.STRING, allowNull: false }, otp: { type: DataTypes.STRING(6), allowNull: false }, expires_at: { type: DataTypes.DATE, allowNull: false }, used: { type: DataTypes.BOOLEAN, defaultValue: false }, } // AuthService.storeOtp() async storeOtp(email, otp) { await EmailOtp.destroy({ where: { email } }); // remove old await EmailOtp.create({ email, otp: await bcrypt.hash(otp, 10), // hash before storing expires_at: new Date(Date.now() + 15 * 60 * 1000), }); }
CRITICAL

BUG-004 — No File Type Validation Before Cloudinary Upload

✓ FIXED
File: src/routes/ProductRoutes.js, FeedbackRoutes.js  |  Impact: Malware Upload, SSRF

Multer is configured but only checks file size — it does not validate MIME type or file extension. An attacker can upload an HTML file, an SVG with embedded JavaScript, or an executable disguised with an image extension. The file is streamed directly to Cloudinary without sanitization, and the returned URL is stored in the database and served to users.

Fix: Add a MIME type whitelist in Multer's fileFilter.
// config/multer.js const ALLOWED_MIME_TYPES = ['image/jpeg', 'image/jpg', 'image/png', 'image/webp']; const fileFilter = (req, file, cb) => { if (ALLOWED_MIME_TYPES.includes(file.mimetype)) { cb(null, true); } else { cb(new AppError('Only JPEG, PNG, and WebP images are allowed.', 400), false); } }; const upload = multer({ storage: multer.memoryStorage(), limits: { fileSize: 5 * 1024 * 1024 }, // 5MB fileFilter, });
CRITICAL

BUG-005 — Helmet Security Headers Missing

✓ FIXED
File: src/index.js  |  Impact: XSS, Clickjacking, MIME Sniffing

The Express app has no security headers. Without helmet, the API is missing: Content-Security-Policy, X-Frame-Options (clickjacking), X-Content-Type-Options (MIME sniffing), Strict-Transport-Security, and Referrer-Policy. Modern browsers and security scanners will flag this immediately.

Fix: Install and apply helmet as the first middleware.
// npm install helmet const helmet = require('helmet'); // In src/index.js — add as FIRST middleware app.use(helmet()); app.use(helmet.contentSecurityPolicy({ directives: { defaultSrc: ["'self'"], imgSrc: ["'self'", "https://res.cloudinary.com"], } }));
CRITICAL

BUG-006 — No Token Revocation / Blacklist Mechanism

File: src/middleware/authMiddleware.js, AuthService.js  |  Impact: Compromised Accounts Can't Be Locked

JWTs are stateless — once issued, they remain valid until expiry. If a customer's account is compromised, an admin has no way to invalidate existing tokens. Logout does not invalidate the token server-side. If JWT_SECRET is leaked, all user tokens are valid indefinitely until the secret is rotated.

Fix: Implement a token version counter in the Customer model. On password change or forced logout, increment the counter. Validate the counter during JWT verification.
// Customer model: add token_version field token_version: { type: DataTypes.INTEGER, defaultValue: 0 } // When issuing JWT, embed token_version: const token = jwt.sign( { id: customer.id, tokenVersion: customer.token_version }, process.env.JWT_SECRET, { expiresIn: '7d' } ); // In authMiddleware, validate version: if (decoded.tokenVersion !== user.token_version) { return next(new AppError('Session expired. Please log in again.', 401)); } // On password change or admin force-logout: await customer.increment('token_version');
CRITICAL

BUG-007 — COD Block Not Enforced for Authenticated Guest Paths

File: src/controllers/OrderController.js  |  Impact: Fraud — Blocked Customers Place COD Orders

The system implements a COD fraud-prevention system (blocks customers after RTO events). However, the order creation endpoint supports both authenticated and guest flows. The COD block check uses a soft check — if customer.is_cod_blocked is true, it logs a warning but does not consistently throw an error in all code paths. A determined user can bypass by using a slightly different email/phone combination.

Fix: Hard-check COD block at the very beginning of order creation, before any DB writes. Also check by contact (email + phone combination) not just customer ID.
Section 5
High & Medium Security Issues
HIGH

BUG-008 — Password Policy Too Weak (Min 6 Characters)

✓ FIXED
File: src/controllers/AuthController.js

Passwords only require 6+ characters with no complexity rules. Industry standard is 8–12 minimum with uppercase, number, and symbol requirements.

Fix: Enforce: min 8 chars, 1 uppercase, 1 number, 1 special character using regex validation.
HIGH

BUG-009 — Error Messages Leak Database Schema Details

✓ FIXED
File: src/utils/http.js, multiple controllers

Error responses in some paths include raw Sequelize error messages like SequelizeUniqueConstraintError: duplicate key value violates unique constraint "customers_email_key". This reveals table names and column names to attackers.

Fix: Catch Sequelize error types in the global errorHandler and return user-friendly messages. Never leak stack traces or DB errors in production.
// src/middleware/errorHandler.js if (err.name === 'SequelizeUniqueConstraintError') { return res.status(409).json({ error: 'This email or phone is already registered.' }); } if (err.name === 'SequelizeValidationError') { return res.status(400).json({ error: 'Invalid input data.' }); }
HIGH

BUG-010 — Admin Middleware Does Not Check Token Expiry on Admin Routes

File: src/middleware/authMiddleware.js

Admin JWT tokens may have the same expiry as customer tokens. Admin sessions should be much shorter (e.g., 1 hour) and require re-authentication for sensitive operations.

Fix: Separate admin token lifetime (1h) from customer tokens (7d). Add IP allowlist for admin endpoints if admins operate from known locations.
MEDIUM

BUG-011 — Razorpay Webhook Not Verified

File: src/routes/RazorpayRoutes.js, RazorpayController.js

Payment verification is done client-side (frontend calls verify-payment). If Razorpay sends webhook events (payment.captured, order.paid), there is no webhook endpoint or signature verification for server-side payment confirmation. Client-side payment verification can be spoofed if the frontend is compromised.

Fix: Add a Razorpay webhook endpoint that verifies the X-Razorpay-Signature header and processes payment.captured events server-side as the source of truth.
MEDIUM

BUG-012 — CORS Origin from Environment Not Validated

File: src/index.js

CORS origins are stored in a comma-separated env variable. If the env var is misconfigured (extra space, typo), it silently allows wrong origins. There is no validation of the origin format.

Fix: Validate each CORS origin is a valid URL. Log a startup warning if origins list is empty or malformed.
MEDIUM

BUG-013 — Referral Code Predictable (UUID-based but short)

File: src/services/AuthService.js

Referral codes are generated as uppercase substrings of a UUID. Short codes (6-8 characters) are guessable by brute force, allowing attackers to claim rewards by trying codes in bulk from throwaway accounts.

Fix: Enforce one referral code claim per phone number verified by OTP. Add a limit on referral code applications per IP address.
Section 6
Order & Payment Business Logic Bugs
Bugs that cause incorrect financial outcomes or broken order flows
HIGH

BUG-014 — Razorpay Order Amount Race Condition

File: src/controllers/RazorpayController.js, OrderController.js

The Razorpay order is created with an amount, but the final order amount is calculated separately during createOrder(). If prices or coupon values change between Razorpay order creation and payment verification (possible in multi-tab or slow-network scenarios), the paid amount may not match the order amount. The system does not cross-validate amounts.

Fix: Store the Razorpay order ID along with the expected amount at order creation time. During verifyPayment(), re-fetch the Razorpay order via API and verify the paid amount matches the expected amount — reject if there is any discrepancy.
HIGH

BUG-015 — Order Cancellation Window Allows 24h But Shipped Orders Can Be Cancelled

File: src/controllers/OrderController.js

The cancellation logic checks a 24-hour window from order creation. However, orders can be shipped within 24 hours. If an order is already pushed to ShipRocket and an AWB assigned, the customer-facing cancel may succeed in the database but leave a live shipment in transit with no automatic ShipRocket cancellation triggered.

Fix: During cancellation, check the shipping status. If AWB is assigned, automatically trigger ShipRocket cancellation API before updating the database order status.
// OrderController.cancelOrder() — add shipping cancellation if (order.shiprocket_order_id && order.awb_code) { await ShipRocketService.cancelShipment(order.shiprocket_shipment_id); } // Then update DB status
HIGH

BUG-016 — Refund Not Triggered on Payment Order Cancellation

✓ FIXED
File: src/controllers/OrderController.js

When a prepaid order is cancelled, the system marks the order as cancelled and credits the wallet. However, there is no Razorpay refund API call initiated. The customer's money stays with the business and only a wallet credit is given — this may not comply with consumer protection regulations and will cause customer disputes.

Fix: On cancellation of a prepaid order, call razorpay.payments.refund(paymentId, { amount }) to initiate an actual bank refund. The wallet credit should be optional, not the only refund path.
// OrderController.cancelOrder() if (order.payment_method !== 'COD' && order.razorpay_payment_id) { const refundAmount = calculateRefundAmount(order); // in paise await razorpay.payments.refund(order.razorpay_payment_id, { amount: refundAmount, notes: { reason: 'Customer cancellation', orderId: order.id } }); order.refund_status = 'initiated'; }
HIGH

BUG-017 — Duplicate Order Possible on Network Retry

✓ FIXED
File: src/controllers/OrderController.js

If the client times out during createOrder() and retries the request, there is no idempotency key check. The same Razorpay payment ID could be used to create two orders, or two pending orders could be created for the same cart. The transaction rollback protects DB integrity but does not prevent duplicate order creation from retried requests.

Fix: Add a unique constraint on razorpay_payment_id in the Order table. Also check if an order already exists with the given payment ID before creating a new one.
MEDIUM

BUG-018 — Coupon Discount Applied Before Stock Validation

File: src/controllers/OrderController.js

Coupon validation and discount calculation happens early in createOrder(), but stock is checked later. If stock validation fails midway and the transaction rolls back, the coupon usage counter has already been incremented (if implemented before the rollback point).

Fix: Ensure coupon usage increment is inside the database transaction and only committed after all stock deductions succeed.
MEDIUM

BUG-019 — Wallet Deduction Without Lock Causes Double-Spend

✓ FIXED
File: src/services/WalletService.js

The wallet deduction reads the balance, verifies sufficiency, then updates in separate steps. Without a row-level lock (SELECT FOR UPDATE), two concurrent requests can both read the same balance, both validate it as sufficient, and both deduct — resulting in a negative wallet balance.

Fix: Use Sequelize's lock: transaction.LOCK.UPDATE when reading wallet balance inside a transaction.
// WalletService.debitWallet() const customer = await Customer.findByPk(customerId, { transaction, lock: transaction.LOCK.UPDATE, // prevents concurrent reads }); if (customer.wallet_balance < amount) { throw new AppError('Insufficient wallet balance', 400); } await customer.decrement('wallet_balance', { by: amount, transaction });
Section 7
Inventory & Stock Management Bugs
HIGH

BUG-020 — JSONB Color Stock Update Is Not Atomic

File: src/controllers/OrderController.js

Product inventory is stored as a JSONB field (color_stocks). Decrementing stock requires: read JSONB → modify in JS → write back. This read-modify-write on a JSONB field inside a transaction with row lock is safe for single-row updates, but if two orders for the same product and color are placed simultaneously, the second transaction will wait on the row lock — but the JSONB patch logic in JavaScript may not correctly handle the version that was updated by the first transaction.

Fix: Use a PostgreSQL JSONB update expression to do atomic decrement inside the database, or migrate color-specific stock to a separate product_variants table with proper integer columns and row-level locks.
// Recommended: PostgreSQL native JSONB update (atomic) await sequelize.query(` UPDATE products SET color_stocks = jsonb_set( color_stocks, '{${colorKey}}', (COALESCE((color_stocks->>'${colorKey}')::int, 0) - :qty)::text::jsonb ) WHERE id = :productId AND (color_stocks->>'${colorKey}')::int >= :qty `, { replacements: { colorKey, qty, productId }, transaction });
HIGH

BUG-021 — Stock Not Restocked on Payment Failure

File: src/controllers/OrderController.js

During prepaid order flow, stock is decremented when the order is created (before payment). If the Razorpay payment fails or is abandoned by the customer, the order may sit in pending_payment state indefinitely with stock locked. There is no timeout or cleanup job that restores stock for unpaid orders.

Fix: Add a cron job that runs every 30 minutes to find orders in pending_payment status older than 30 minutes, cancels them, and restores inventory.
// server.js — add cleanup cron cron.schedule('*/30 * * * *', async () => { const staleOrders = await Order.findAll({ where: { status: 'pending_payment', createdAt: { [Op.lt]: new Date(Date.now() - 30 * 60 * 1000) } } }); for (const order of staleOrders) { await restoreInventory(order); await order.update({ status: 'payment_failed' }); } });
MEDIUM

BUG-022 — No Low-Stock Threshold Alerts

File: src/services/ProductService.js (missing)

Product model has a low_stock_threshold field, but there is no logic that sends alerts when stock drops below this threshold. Admins have no visibility into which products are running out until they manually check the dashboard.

Fix: After each successful stock decrement, check if remaining stock is ≤ threshold and send an email alert to the admin email.
Section 8
Wallet & Referral System Bugs
HIGH

BUG-023 — Referral Abuse: Multiple Accounts, One Phone Number

File: src/services/AuthService.js, WalletService.js

A user can create multiple accounts with different emails but reuse the same phone number (if phone uniqueness is not enforced), apply a referral code on each, and earn the referrer unlimited bonus credits. Even with phone uniqueness, email-only accounts can be created in bulk from disposable email services.

Fix: (1) Enforce phone number uniqueness and verify via OTP during registration. (2) Allow each referral code to be used only once per device fingerprint or phone number. (3) Cap max referral earnings per referrer per month.
MEDIUM

BUG-024 — Referral Reward Credited Even If Order Is Returned

✓ FIXED
File: src/services/WalletService.js

The referral order reward is scheduled 7 days after delivery confirmation. The cron job (processDuePendingCredits()) processes all due credits without checking if the triggering order was subsequently returned or refunded. A user can place an order, trigger a referral reward, then immediately initiate a return.

Fix: Before releasing a pending referral credit, check the order status. If the order is in returned/refunded state, cancel the pending credit instead of releasing it.
// WalletService.processDuePendingCredits() for (const credit of dueCredits) { if (credit.order_id) { const order = await Order.findByPk(credit.order_id); if (['returned', 'refunded', 'cancelled'].includes(order.status)) { await credit.update({ status: 'cancelled' }); continue; // skip — don't reward returned orders } } await creditNow(credit); // safe to release }
MEDIUM

BUG-025 — Wallet Balance Can Go Negative

✓ FIXED
File: src/services/WalletService.js

There is no database-level constraint preventing wallet_balance from going negative. If the race condition in BUG-019 occurs, or if a refund calculation error results in excess credit, the balance could be corrupted. Application-layer checks alone are insufficient.

Fix: Add a CHECK constraint at the database level: ALTER TABLE customers ADD CONSTRAINT wallet_balance_non_negative CHECK (wallet_balance >= 0);
Section 9
Cart & Coupon System Bugs
HIGH

BUG-026 — Cart Items Not Validated Against Current Stock at Checkout

✓ FIXED
File: src/services/CartService.js, OrderController.js

When a customer adds an item to cart, the stock is checked at that moment. However, between adding to cart and checkout (which could be days later), another customer may have purchased the last unit. The cart shows the item as available but checkout fails partway through with a cryptic database error rather than a clear "item out of stock" message.

Fix: At the start of createOrder(), re-validate every cart item against current stock. Return a clear error per out-of-stock item before touching payment or shipping systems.
HIGH

BUG-027 — Coupon per-User Limit Not Enforced Atomically

✓ FIXED
File: src/services/CouponService.js

The per-user coupon usage limit is validated by counting how many times the customer used the coupon in past orders. This check happens before the order is created. In a concurrent scenario (two checkout tabs open simultaneously), both can pass the limit check and both orders use the coupon, exceeding the per-user limit.

Fix: Add coupon usage increment inside the order creation transaction with a SELECT FOR UPDATE lock on the coupon row.
MEDIUM

BUG-028 — Cart Has No Expiry / Guest Cart Not Merged

File: src/services/CartService.js

Cart items can sit indefinitely. Old carts from months ago with out-of-stock or discontinued products accumulate. There is also no mechanism to merge a guest cart into a logged-in user's cart on login — a common expected behavior in e-commerce.

Fix: (1) Add a cron job to clean carts older than 30 days. (2) Implement guest cart (local storage on frontend) merged into user cart on login via an API endpoint.
Section 10
Database Performance Issues
Issues that will cause slowdowns under real user traffic
HIGH

BUG-029 — No Database Indexes on Critical Query Columns

✓ FIXED
File: All model files

Sequelize models define no indexes. Full-table scans will occur on: customers.email, customers.phone, orders.customer_id, orders.status, wallet_transactions.customer_id, products.slug, coupons.code. These are the most frequently queried columns.

Fix: Add indexes in model definitions or as a migration.
// In Customer model: indexes: [ { unique: true, fields: ['email'] }, { unique: true, fields: ['phone'] }, { fields: ['referral_code'] }, ] // In Order model: indexes: [ { fields: ['customer_id'] }, { fields: ['status'] }, { fields: ['razorpay_payment_id'] }, { fields: ['shiprocket_order_id'] }, ] // In Product model: indexes: [ { unique: true, fields: ['slug'] }, { fields: ['is_active', 'is_archived'] }, // composite for listing ]
HIGH

BUG-030 — No Pagination on List Endpoints

✓ FIXED
File: ProductController.js, OrderController.js, WalletController.js

Product listing, order history, and wallet transactions load all records from the database into memory. With 10,000 products or 100,000 orders, this will OOM-crash the server and time out every user request.

Fix: Add page and limit query params with Sequelize offset/limit.
// ProductController.getProducts() — add pagination const page = parseInt(req.query.page) || 1; const limit = Math.min(parseInt(req.query.limit) || 20, 100); const offset = (page - 1) * limit; const { count, rows } = await Product.findAndCountAll({ where: filters, limit, offset, order: [['createdAt', 'DESC']], }); return ok(res, { products: rows, pagination: { total: count, page, limit, pages: Math.ceil(count / limit) } });
HIGH

BUG-031 — Runtime Column Creation (No Migration System)

File: src/utils/orderLifecycle.js, feedbackSchema.js, orderItemActions.js

On every server startup, the app calls ensureColumns() functions that add missing DB columns via queryInterface.addColumn(). This is fragile: (1) multiple instances starting simultaneously race to add the same column, (2) there is no rollback mechanism if something goes wrong, (3) it makes deployment unpredictable.

Fix: Replace with proper Sequelize CLI migrations. Run migrations as a pre-startup step (npx sequelize-cli db:migrate).
MEDIUM

BUG-032 — N+1 Query Pattern in Order List

File: src/controllers/OrderController.js

Fetching a list of orders and then iterating to fetch associated order items or product details in a loop creates N+1 queries. 50 orders = 51 database queries instead of 1.

Fix: Use Sequelize include with proper eager loading. Specify only required attributes to avoid loading entire rows.
Section 11
API Performance Issues
MEDIUM

BUG-033 — No Response Compression

✓ FIXED

API responses are not gzip-compressed. A product catalog response can be 100KB+ uncompressed. With compression, this shrinks to ~15KB — a 7× reduction in bandwidth and faster load times.

Fix: npm install compression and add as middleware before routes.
MEDIUM

BUG-034 — Synchronous Cloudinary Uploads Block Requests

✓ FIXED

Product image uploads wait for Cloudinary's response before sending API response. Uploading 12 product images can take 15–30 seconds, blocking the admin request thread.

Fix: Upload images to Cloudinary in parallel using Promise.all() or use a queue (Bull) to process uploads asynchronously.
// Parallel upload (quick win) const uploadedImages = await Promise.all( files.map(file => cloudinary.uploader.upload(file.path, { folder: 'products' })) );
MEDIUM

BUG-035 — No Caching for Catalog Endpoints

✓ FIXED

Product lists, color/material/variety/occasion taxonomies are fetched from DB on every request. These rarely change but are fetched hundreds of times per minute during peak traffic. No HTTP caching headers (ETag, Cache-Control) are set either.

Fix: Set Cache-Control: public, max-age=300 on catalog endpoints. Use Redis to cache taxonomy data with 5-minute TTL.
LOW

BUG-036 — Cron Jobs Not Cluster-Safe

If the backend scales to 2+ instances on Render.com, both instances will run the referral payout cron simultaneously, potentially double-crediting wallets (though deduplication keys partially mitigate this).

Fix: Use a distributed lock (Redis SETNX) or move cron to a separate worker process.
